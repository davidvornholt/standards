# Standards release tooling

This private workspace owns the repository-only npm and GitHub release boundary. It is not part of the published `@davidvornholt/standards` package.

Release packing treats `packages/standards-cli` as caller-owned and read-only and requires the exact Bun 1.3.14 artifact format. Read-only namespace inspection refuses any pre-existing `SOURCE_COMMIT` entry, including non-regular files. The unmodified source package is packed with scripts disabled, then the release boundary accepts only Bun's regular-file USTAR dialect: exact magic and version, valid checksums and field terminators, zero header/content padding, and exactly two terminal zero blocks. It deterministically rewrites `package/package.json` so `gitHead` equals the tested commit before appending the matching `package/SOURCE_COMMIT`; every other public entry and its semantics remain unchanged. The completed artifact is re-read through the same identity verifier used for existing npm artifacts.

Every successful quality run whose event payload identifies a default-branch push enters the publisher and performs the cheap manifest checks before dependency installation and npm inspection. Immediately before npm publication, the same tested mutation owner freshly reads the repository's live default branch, proves through GitHub's comparison API that the release SHA remains its ancestor, and spawns npm with the trusted-publishing OIDC environment but without the GitHub API token. The GitHub reconciler independently repeats that authorization immediately before each tag or Release mutation, so release-state scans and queued branch changes cannot make its proof stale. No first-parent comparison gates inspection. If the current version is absent, the tested commit is packed and published. If it exists, the registry tarball's SRI and strict tar identity are verified before its `gitHead` and matching `SOURCE_COMMIT` are treated as an untrusted candidate. The candidate must be an available ancestor of the current tested commit. Current tested tooling materializes that candidate in a hooks-disabled detached temporary worktree, installs and runs none of its code or scripts, reproduces its package, and requires both SRI and byte-for-byte equality with npm before emitting the source commit for GitHub tag and Release reconciliation. Scoped cleanup removes the temporary worktree on success, failure, or interruption. This makes failed or coalesced bump runs and npm-success/GitHub-failure retries recoverable while legacy, self-asserted, mismatched, or malformed artifacts fail closed.

The default-branch quality and publish boundaries use GitHub's non-replacing `queue: max` semantics. Pull-request quality runs retain cancellation and the required `check` identity, while default-branch pushes use `main-check`. GitHub queues by wait time rather than commit history and does not guarantee dispatch order, so an older unpublished version that reaches the publisher after a newer one fails closed instead of moving npm latest backwards.

## Configuration

- **`GITHUB_REPOSITORY`** (required by GitHub release commands) — repository identifier in `owner/repository` form. GitHub Actions supplies it automatically.
- **`GH_TOKEN` / `GITHUB_TOKEN`** (one required by GitHub release commands) — GitHub API token used to inspect and reconcile tags and releases. The first non-empty value is used, preferring `GH_TOKEN`; the release workflow exposes its GitHub-provided token as `GH_TOKEN`.
- **`ACTIONS_ID_TOKEN_REQUEST_TOKEN`** (required secret for npm provenance publication; no local default) — short-lived bearer credential supplied automatically by GitHub Actions when the publishing job grants `id-token: write`. npm uses it to request the OIDC identity token for trusted publication. Treat it as secret: do not log, persist, or expose it outside the publishing subprocess.
- **`ACTIONS_ID_TOKEN_REQUEST_URL`** (required non-secret configuration for npm provenance publication; no local default) — GitHub Actions supplies the OIDC request endpoint consumed by npm during trusted publication. The URL does not authorize a request without `ACTIONS_ID_TOKEN_REQUEST_TOKEN`, so it is configuration rather than a secret.
