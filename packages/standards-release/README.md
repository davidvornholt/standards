# Standards release tooling

This private workspace owns the repository-only npm and GitHub release boundary. It is not part of the published `@davidvornholt/standards` package.

Release packing treats `packages/standards-cli` as caller-owned and read-only and requires the exact Bun 1.3.14 artifact format. Read-only namespace inspection refuses any pre-existing `SOURCE_COMMIT` entry, including non-regular files. The unmodified source package is packed with scripts disabled, then the release boundary accepts only Bun's regular-file USTAR dialect: exact magic and version, valid checksums and field terminators, zero header/content padding, and exactly two terminal zero blocks. It deterministically rewrites `package/package.json` so `gitHead` equals the tested commit before appending the matching `package/SOURCE_COMMIT`; every other public entry and its semantics remain unchanged. The completed artifact is re-read through the same identity verifier used for existing npm artifacts.

Every successful quality run whose event payload identifies a default-branch push enters the publisher and performs the cheap manifest checks before dependency installation and npm inspection. If the current version is absent, the tested commit is packed, then its source artifact bytes are staged into a private mode-`0400` file, opened through a read-only descriptor, and unlinked so no pathname can select a replacement generation. The publisher re-reads that exact staged generation and verifies its expected SHA-512 integrity and embedded source identity before preparing a process-private `.tgz` adapter. With the descriptor and adapter fixed, the same tested mutation owner freshly reads the repository's live default branch and proves through GitHub's comparison API that the release SHA remains its ancestor; this authorization is the last awaited operation before the npm subprocess starts. npm receives the descriptor as an inherited file descriptor and consumes it only through the private `.tgz` adapter, with the trusted-publishing OIDC environment but without either GitHub API token. Scoped finalization closes the descriptor and unlocks and removes only the publisher's private adapter directory on success, failure, or interruption. The GitHub reconciler independently repeats live default-branch authorization immediately before each tag or Release mutation. Before the Release POST, it then reads and peels the exact tag as the final awaited authorization and requires that tag to resolve to the expected commit. Canonical GitHub settings allow creation of `v*` tags but prohibit every later update or deletion, while immutable releases protect the published release and its associated tag. Together these controls keep release-state scans, queued branch changes, and concurrent tag replacement from making the mutation proof stale. No first-parent comparison gates inspection. If the version already exists, the registry tarball's SRI and strict tar identity are verified before its `gitHead` and matching `SOURCE_COMMIT` are treated as an untrusted candidate. The candidate must be an available ancestor of the current tested commit. Current tested tooling materializes that candidate in a hooks-disabled detached temporary worktree, installs and runs none of its code or scripts, reproduces its package, and requires both SRI and byte-for-byte equality with npm before emitting the source commit for GitHub tag and Release reconciliation. Separately scoped cleanup removes the temporary worktree on success, failure, or interruption. This makes failed or coalesced bump runs and npm-success/GitHub-failure retries recoverable while legacy, self-asserted, mismatched, or malformed artifacts fail closed.

The default-branch quality and publish boundaries use GitHub's non-replacing `queue: max` semantics. Pull-request quality runs retain cancellation and the required `check` identity, while default-branch pushes use `main-check`. GitHub queues by wait time rather than commit history and does not guarantee dispatch order, so an older unpublished version that reaches the publisher after a newer one fails closed instead of moving npm latest backwards.

## Configuration

- **`GITHUB_REPOSITORY`** (required by GitHub release commands) — repository identifier in `owner/repository` form. GitHub Actions supplies it automatically.
- **`GH_TOKEN` / `GITHUB_TOKEN`** (one required by GitHub release commands) — GitHub API token used to inspect and reconcile tags and releases. The first non-empty value is used, preferring `GH_TOKEN`; the release workflow exposes its GitHub-provided token as `GH_TOKEN`.
- **`ACTIONS_ID_TOKEN_REQUEST_TOKEN`** (required secret for npm provenance publication; no local default) — short-lived bearer credential supplied automatically by GitHub Actions when the publishing job grants `id-token: write`. npm uses it to request the OIDC identity token for trusted publication. Treat it as secret: do not log, persist, or expose it outside the publishing subprocess.
- **`ACTIONS_ID_TOKEN_REQUEST_URL`** (required non-secret configuration for npm provenance publication; no local default) — GitHub Actions supplies the OIDC request endpoint consumed by npm during trusted publication. The URL does not authorize a request without `ACTIONS_ID_TOKEN_REQUEST_TOKEN`, so it is configuration rather than a secret.
