# @davidvornholt/standards

CLI for bootstrapping, synchronizing, and validating repositories that consume [davidvornholt/standards](https://github.com/davidvornholt/standards).

The sync engine accepts only real directories and regular files in managed and seed trees. It rejects symlinks and special nodes in source or consumer paths and preflights the complete filesystem plan before writing or deleting anything. For an initialized consumer, bare sync validates `sync-standards.lock` before source selection and uses its `upstream`; the managed `sync-standards.json` can therefore be repaired instead of selecting its own replacement. Only an explicit `--from` overrides locked source authority, while a lockless legacy consumer falls back to its managed manifest. The lock persists every observed repository-owned seed path, and sync rejects implicit seed-to-managed or managed-to-seed ownership changes before mutation. Legacy locks are migrated with the contract-v1 seed baseline so selecting an older or malformed source cannot silently erase established seed ownership.

The CLI supports filesystem-mutating commands only on Linux, so package installation is restricted to Linux. Safe publication resolves `renameat2` from the libc already mapped into the current Bun process, whether glibc or musl, and requires a kernel/filesystem combination that supports `RENAME_NOREPLACE`; the CLI fails closed when that loaded libc or atomic primitive is unavailable. Its transaction contract requires one standards writer and a stable managed tree: while `init`, `sync`, or recovery runs, no other process may create, edit, delete, rename, replace, or remount a managed target or any of its ancestor directories, including the repository root. Managed outputs and seed targets must not claim the CLI-owned `sync-standards.lock` or this reserved transaction artifact grammar: fixed names `.standards-transaction`, `.standards-transaction-cleanup`, `.standards-transaction-owner-reservation`, and `.standards-transaction-reservation`; prefix families `.standards-transaction-publication-*`, `.standards-owner-publication-*`, `.standards-parent-*`, and `.standards-removal-*`; and lowercase UUID-v4 atomic tails `.standards-transaction-reservation.<uuid-v4>.tmp`, `OWNER.<uuid-v4>.tmp`, and `.standards-parent-binding-<transaction-uuid-v4>-<index>.<write-uuid-v4>.tmp`. `.standards-transaction-user` does not match this grammar and remains consumer-visible and unreserved. Only those fixed names, prefix families, and lowercase UUID-v4 atomic tails are exclusively CLI-owned; another process or user must not create or modify them while a transaction or recovery journal exists. Atomic-tail recovery requires lowercase canonical UUID-v4 values and a canonical decimal index; other tail lookalikes are not adopted by that recovery grammar. During owner publication, the durable root token records the partial `OWNER` inode before any owner bytes are written; recovery removes that tail only while the inode still matches and otherwise retains it for inspection. The CLI fails closed when a reserved fixed entry already exists before mutation. A staged publication without an inode-bound owner token is retained for manual inspection because recovery cannot prove ownership. It detects interference at several boundaries and retains its recovery journal when it can, but this contract is not a cryptographic defense against the same user forging an exact reserved artifact and does not claim atomicity against an uncooperative actor racing the final verification and commit marker.

Cleanup never unlinks a replaceable pathname. It atomically moves the exact verified inode into fixed-size `.standards-removal-<sha256>.entry` quarantine and retains the matching `.json` and `.tail` ownership record, including after a transaction reaches a terminal state. UUID-suffixed `.draft.*` records are also retained: a partial draft is ignored safely after a crash, while a complete draft remains a hard link to the inode published as the ownership record. Before mutating or recovering a Git worktree, the CLI installs an owned, last-wins block for only those reserved recovery artifact forms in the repository-local Git exclusion file; it fails closed on linked or replaced Git metadata, leaves non-reserved lookalikes such as `.standards-transaction-user` visible, and therefore keeps the durable history out of ordinary `git status` and `git add -A`. Safe exclusion replacement may retain verified `exclude.standards-*.tmp` generations inside Git common metadata. This durable fail-closed history can consume storage over time. The CLI deliberately provides no automatic garbage collection because Linux does not provide an identity-conditional unlink primitive; deleting a pathname after a separate identity check would recreate the race the quarantine prevents. Do not remove these artifacts by wildcard. If storage must be reclaimed, first stop every standards writer and recovery process, back up the repository, verify the public repository state, and inspect each ownership-record generation before manually removing that exact terminal generation; removing an active or unverified generation can make recovery fail closed.

```sh
bunx @davidvornholt/standards init
standards sync
standards check
standards doctor
standards github --check
standards github --apply
```

`github --check` compares the live GitHub repository against the merged declaration in `.github/settings.json` + `.github/settings.local.json`; `github --apply` converges the declared repository settings, canonical classic branch protection on the repository's actual default branch, and the exact repository-wide ruleset set. The `repository` object accepts exactly 7 keys: `allow_auto_merge`, `allow_merge_commit`, `allow_rebase_merge`, `allow_squash_merge`, `delete_branch_on_merge`, `squash_merge_commit_message`, and `squash_merge_commit_title`. Any other repository key fails before any API request. The canonical declaration currently owns all 7 repository keys, so `.github/settings.local.json` keeps `repository` empty and currently extends only `rulesets` and `environments`; declaring a canonically owned repository key locally is a collision that fails instead of overriding canonical state. Local settings cannot declare or override `default_branch_protection`. Declared rulesets are active branch rulesets with explicit ref-name conditions, no bypass actors, and at least one unique supported rule (`deletion`, `non_fast_forward`, `required_linear_history`, `pull_request`, or `required_status_checks`); unknown fields and malformed rule parameters fail before any API request. Repository identity and lifecycle are outside the settings contract: both files reject `default_branch`, `name`, `private`, `visibility`, `archived`, and `is_template` before any API request. Classic protection declarations must set `required_signatures` to `false`; enabling signatures is unsupported. Check first proves from the public branch summary that the default branch is protected by classic branch protection, then compares the exact admin-only details when the token can read them; a non-admin 401, 403, or concealment 404 makes only those details unverifiable, while missing classic protection remains drift. Every present live repository merge setting must be a boolean or exact supported squash title/message enum; malformed values fail both commands. Check reports an absent managed repository setting as unverifiable for a non-admin token, while apply requires every declared setting to be admin-visible and fails before any write when one is absent. GitHub may omit or return null for disabled `required_status_checks` and `required_pull_request_reviews`; check treats those states as disabled policy and reports drift when the canonical policy enables them, while malformed present objects fail closed. Apply requires readable details, validates every managed live resource before writing, repairs disabled optional sections, freshly re-reads and validates the repository's default-branch identity after a protection update, verifies that branch's classic protection, and only then deletes undeclared rulesets. For environments, both commands reconcile the exact protection and protected-branch policy of each declared environment name. Declarations must set `deployment_branch_policy` to `{"protected_branches":true,"custom_branch_policies":false}` and must not contain `deployment_branch_policies`; custom branch and tag policies are outside this contract. A live custom mode is drift, and apply replaces it with one environment update without branch-policy requests. Only the `wait_timer`, `required_reviewers`, and single `branch_policy` protection rule types are supported; unknown, duplicate, missing, or inconsistent branch-policy rules fail closed. Custom GitHub App deployment protection gates are intentionally unsupported declaration state: check reports every enabled gate as undeclared drift and apply disables it only after the environment protection update succeeds. Environment names are case-insensitive and limited to 255 characters; `wait_timer` is an integer from 0 through 43,200, and `reviewers` contains at most six users or teams identified by positive safe integer IDs. Invalid declarations fail before any API request. Undeclared environments are unmanaged: the CLI neither lists nor deletes them because they can own scoped secrets. The protected `standards-sync` environment admits branches covered by classic protection; ruleset-only enforcement does not qualify. The schedule and repository dispatch bind the workflow to the repository's default branch, while required-check producers use the live default-branch identity so renames do not strand required checks. Secret values remain separate and are never listed, read, or written by either command. `check` runs the same comparison whenever `.github/settings.json` is present.

Immediately before every undeclared ruleset deletion, apply freshly reads and decodes the live repository default branch, proves its public classic-protection summary and admin-only protection detail still match the declaration, and re-reads the exact ruleset ID. The deletion proceeds only when that ruleset's ID and undeclared name remain unchanged; branch, protection, identity, rename, and response-contract races fail before the destructive request.

## Configuration

- **`sync-standards.local.json`** (optional for existing consumers) — checked-in repository policy with required `ref` and `scheduledSync` fields. Missing files default to `{"ref":"refs/heads/main","scheduledSync":true}`. `ref` accepts only a qualified branch, qualified tag, or full commit SHA. Setting `scheduledSync` to `false` skips scheduled GitHub Actions syncs but not default-branch `standards-sync` repository dispatches or local runs; the weekly workflow cadence remains canonical. The installed preflight requires `@davidvornholt/standards` >=0.5.0 declared as an exact direct development dependency for every policy, including the default. The checked-in explicit-ESM preflight bundle validates that declaration before dependency setup. Fetched sources must declare `"syncPolicyContractVersion": 1` in `sync-standards.json`; pre-contract refs are rejected before mirroring. Contract-v1 sources must not manage the repository-owned control seams `.github/settings.local.json`, `AGENTS.local.md`, `biome.jsonc`, or `sync-standards.local.json`.
- **`GITHUB_EVENT_NAME`, `GITHUB_OUTPUT`, and `GITHUB_WORKSPACE`** (required for the generated preflight action; no defaults) — GitHub Actions supplies all three. `GITHUB_EVENT_NAME` accepts only `schedule` or `repository_dispatch`; scheduled runs honor `scheduledSync`, while repository dispatches always request a sync. `GITHUB_OUTPUT` identifies the file where the action writes its `run_sync` output. `GITHUB_WORKSPACE` identifies the consumer repository root where the action reads `package.json` and `sync-standards.local.json`.
- **`GH_TOKEN` / `GITHUB_TOKEN`** (optional for CLI commands) — GitHub API token for the `github` command and the GitHub portion of `check`. When neither is set, those CLI paths try the token from the local `gh` CLI; with no token at all, reads still work on public repositories. `--apply` and private-repo reads need an authenticated token, and repo merge settings are only visible (and thus verifiable) to admin tokens — non-admin runs report them as unverifiable.
- **`STANDARDS_SYNC_ENVIRONMENT_TOKEN`** (optional for the canonical workflow) — fine-grained PAT with Contents, Pull requests, and Workflows repository permissions set to write, stored separately as a secret on the protected `standards-sync` environment. The selected credential is passed to checkout's persisted Git authentication and to `GH_TOKEN`, so pushes and PR creation use the same identity. Workflows write lets a sync push canonical changes under `.github/workflows`; the workflow falls back to `GITHUB_TOKEN`, but pull requests opened with that token need a manual nudge before their workflows run.

## Upgrade from v0.4

Upgrade the consumer-owned CLI dependency before syncing current canonical files: run `bun add --dev --exact @davidvornholt/standards@0.5.0`. Leave `sync-standards.local.json` absent or at its exact default, run a bare `bun standards sync` from the repository's default branch to install the controller and settings, then run `bun standards github --apply` with admin auth. Apply installs and verifies classic protection before deleting the legacy canonical ruleset, preserving the protected-only environment boundary throughout the migration. Only after that succeeds should you pin a non-default ref. Give the workflow PAT Contents, Pull requests, and Workflows repository permissions set to write, move it to the protected `standards-sync` environment as `STANDARDS_SYNC_ENVIRONMENT_TOKEN`, then delete the legacy repository-level `STANDARDS_SYNC_TOKEN` so it cannot be reused outside that environment.

See the standards repository README for the ownership model and adoption workflow.
