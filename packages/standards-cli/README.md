# @davidvornholt/standards

CLI for bootstrapping, synchronizing, and validating repositories that consume [davidvornholt/standards](https://github.com/davidvornholt/standards).

The sync engine accepts only real directories and regular files in managed and seed trees. It rejects symlinks and special nodes in source or consumer paths and preflights the complete filesystem plan before writing or deleting anything.

```sh
bunx @davidvornholt/standards init
standards sync
standards check
standards doctor
standards github --check
standards github --apply
```

`github --check` compares the live GitHub repository against the merged declaration in `.github/settings.json` + `.github/settings.local.json`; `github --apply` converges the declared repository settings, canonical classic branch protection on the repository's actual default branch, and the exact repository-wide ruleset set. Local settings may add rulesets but cannot declare or override `default_branch_protection`. Repository identity and lifecycle are outside the settings contract: both files reject `default_branch`, `name`, `private`, `visibility`, `archived`, and `is_template` before any API request. Classic protection declarations must set `required_signatures` to `false`; enabling signatures is unsupported. Check first proves from the public branch summary that the default branch is protected by classic branch protection, then compares the exact admin-only details when the token can read them; a non-admin 401, 403, or concealment 404 makes only those details unverifiable, while missing classic protection remains drift. GitHub may omit or return null for disabled `required_status_checks` and `required_pull_request_reviews`; check treats those states as disabled policy and reports drift when the canonical policy enables them, while malformed present objects fail closed. Apply requires readable details, validates every managed live resource before writing, repairs disabled optional sections, verifies classic protection after updating it, and only then deletes undeclared rulesets. For environments, both commands reconcile the exact protection and protected-branch policy of each declared environment name. Declarations must set `deployment_branch_policy` to `{"protected_branches":true,"custom_branch_policies":false}` and must not contain `deployment_branch_policies`; custom branch and tag policies are outside this contract. A live custom mode is drift, and apply replaces it with one environment update without reading or mutating branch-policy entries. Only the `wait_timer`, `required_reviewers`, and single `branch_policy` protection rule types are supported; unknown, duplicate, missing, or inconsistent branch-policy rules fail closed. Custom GitHub App deployment protection gates are intentionally unsupported declaration state: check reports every enabled gate as undeclared drift and apply disables it only after the environment protection update succeeds. Environment names are case-insensitive and limited to 255 characters; `wait_timer` is an integer from 0 through 43,200, and `reviewers` contains at most six users or teams identified by positive safe integer IDs. Invalid declarations fail before any API request. Undeclared environments are unmanaged: the CLI neither lists nor deletes them because they can own scoped secrets. The protected `standards-sync` environment admits branches covered by classic protection; ruleset-only enforcement does not qualify. The schedule and repository dispatch bind the workflow to the repository's default branch, while required-check producers use the live default-branch identity so renames do not strand required checks. Secret values remain separate and are never listed, read, or written by either command. `check` runs the same comparison whenever `.github/settings.json` is present.

## Configuration

- **`sync-standards.local.json`** (optional for existing consumers) — checked-in repository policy with required `ref` and `scheduledSync` fields. Missing files default to `{"ref":"refs/heads/main","scheduledSync":true}`. `ref` accepts only a qualified branch, qualified tag, or full commit SHA. Setting `scheduledSync` to `false` skips scheduled GitHub Actions syncs but not default-branch `standards-sync` repository dispatches or local runs; the weekly workflow cadence remains canonical. The installed preflight requires `@davidvornholt/standards` >=0.5.0 declared as an exact direct development dependency for every policy, including the default. The checked-in explicit-ESM preflight bundle validates that declaration before dependency setup. Fetched sources must declare `"syncPolicyContractVersion": 1` in `sync-standards.json`; pre-contract refs are rejected before mirroring. Contract-v1 sources must not manage the repository-owned control seams `sync-standards.local.json`, `AGENTS.local.md`, `biome.jsonc`, or `.github/settings.local.json`.
- **`GH_TOKEN` / `GITHUB_TOKEN`** (optional for CLI commands) — GitHub API token for the `github` command and the GitHub portion of `check`. When neither is set, those CLI paths try the token from the local `gh` CLI; with no token at all, reads still work on public repositories. `--apply` and private-repo reads need an authenticated token, and repo merge settings are only visible (and thus verifiable) to admin tokens — non-admin runs report them as unverifiable.
- **`STANDARDS_SYNC_ENVIRONMENT_TOKEN`** (optional for the canonical workflow) — fine-grained PAT stored separately as a secret on the protected `standards-sync` environment. The workflow falls back to `GITHUB_TOKEN`, but pull requests opened with that token need a manual nudge before their workflows run.

## Upgrade from v0.4

Upgrade the consumer-owned CLI dependency before syncing current canonical files: run `bun add --dev --exact @davidvornholt/standards@0.5.0`. Leave `sync-standards.local.json` absent or at its exact default, run a bare `bun standards sync` from the repository's default branch to install the controller and settings, then run `bun standards github --apply` with admin auth. Apply installs and verifies classic protection before deleting the legacy canonical ruleset, preserving the protected-only environment boundary throughout the migration. Only after that succeeds should you pin a non-default ref. Move the workflow PAT to the protected `standards-sync` environment as `STANDARDS_SYNC_ENVIRONMENT_TOKEN`, then delete the legacy repository-level `STANDARDS_SYNC_TOKEN` so it cannot be reused outside that environment.

See the standards repository README for the ownership model and adoption workflow.
